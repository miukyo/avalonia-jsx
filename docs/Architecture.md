# Architecture

Avalonia JS utilizes a dual-process architecture during development and a unified execution environment at runtime.

## The Frontend (`avalonia-jsx`)

The frontend leverages standard web ecosystem tools like Vite and TypeScript, but instead of outputting DOM elements, it outputs a customized JSON representation of Avalonia controls.

- **Solid.js / JSX**: Provides reactive state management and declarative UI syntax.
- **Vite Plugin**: The project includes a custom Vite plugin (`viteCssStyles`) that parses embedded `<style>` tags. It converts standard CSS (like `:hover` pseudo-classes and transitions) into JSON structures that Avalonia can interpret natively.

## The Backend (`Avalonia.JSBridge`)

The backend is a standard .NET Avalonia application that runs a JavaScript engine internally.

- **Jint Engine**: Executes the compiled JavaScript bundle produced by the frontend.
- **Bridge Classes**: Components like `ControlFactory`, `StyleFactory`, and `AnimationFactory` bridge the gap. They take the JSON representations generated by the JS runtime and reflectively instantiate native Avalonia `.NET` objects (`StackPanel`, `TextBlock`, `DoubleTransition`, etc.).
- **Hot Reload Service**: During development, a file watcher monitors the Vite output directory and triggers the `Jint` engine to re-evaluate the code, instantly updating the Avalonia UI.

## Why are they coupled?

`Avalonia.JSBridge` relies on specific metadata, component types, and JSON schemas emitted by `avalonia-jsx`. They are designed to be used synchronously as part of a single framework offering.
